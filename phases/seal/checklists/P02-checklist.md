#### Self-Audit Gate 2b (T3 — `remote_remap_private`)

- [x] **Optimality** — Considered: (a) single remap primitive parameterised by a `RemapFlags` enum vs. two separate `remote_remap_private` / `remote_remap_shared` functions — chose one primitive because the only differences are two `u64` flag words (`open_flags` + `mmap_flags`) and the bootstrap / attach / staging logic is bit-for-bit identical; duplicating it would double the unsafe surface for zero benefit. (b) Doing a fourth `munmap(bootstrap_page, 4096)` remote syscall vs. leaking the 4 KiB scratch page — chose the leak. Reasoning: the only candidate scratch PCs after the arena remap are (i) `bootstrap_page` itself (the munmap target, would be unmapped by its own syscall so the restore `process_vm_writev` would EFAULT), and (ii) the original libc.text NOP slide (still `r-xp`, so `write_remote`-based `remote_syscall` can't stage the svc+brk there — it respects VMA write bits per `man 2 process_vm_writev`). A second POKEDATA bootstrap would work but doubles the restore complexity; the 4 KiB leak is bounded (seals are rare operator-initiated events, not hot paths) and explicitly documented inline at `seal/arena.rs:340-346`. (c) `RemoteAttach` scope — narrow (detach only) vs. wide (detach + regs + scratch). Chose narrow because (i) `remote_syscall` already restores regs + scratch on its own success path (P01 §7 step 9), and (ii) the bootstrap stage restores its own regs + scratch bytes inline BEFORE any `?` can bail (`seal/arena.rs:246-255`), so the guard only needs to guarantee detach — the narrower the Drop, the fewer surprising Drop-order interactions.
